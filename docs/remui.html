Overview

RemUI builds widget trees on the backend and renders them on Flutter through a widget registry. Backend actions and callbacks are transported as JSON.

Core backend entrypoints are in remui.ts and route usage is in index.ts.

UI DSL

Create widgets

const card = ui('Card', {
  child: ui('Padding', { padding: 12 },
    ui('Text', { text: 'Hello' })
  )
});

Navigation helper

ui('IconButton', {
  icon: ui('Icon', { icon: 'search' }),
  onPressed: nav('/ui/search')
});

You can also open a route as a dialog overlay. When the second argument is 'dialog', the current page stays visible behind it.

ui('IconButton', {
  icon: ui('Icon', { icon: 'open_in_new' }),
  onPressed: nav('/ui/dialogTest', 'dialog')
});

Use navReplace when you want the new route to fully replace the current stack. The Flutter runtime calls pushAndRemoveUntil, so back navigation is not possible after the replacement.

ui('IconButton', {
  icon: ui('Icon', { icon: 'home' }),
  onPressed: navReplace('/ui/main')
});

Chainable condition

equate('index', '0', {
  child: ui('Text', { text: 'Zero' })
}).equate('index', '1', {
  child: ui('Text', { text: 'One' })
});

Initialization

Use RemUI.init to set the backend URL, runtime config, custom widget registers, custom pages, and page action handlers.

RemUI.init(
  "http://localhost:3000",
  {"mediaQuery": true},
  registers: {
    "TextO": (j) => Text(j["text"]?.toString() ?? '')
  },
  pages: {
    ".page_cd_a": (j) => PageCDA(j)
  },
  pageFunction: {
    ".main": (onClick, url) {
      if (url == "/next") {
        RemUI.changePage("/ui/next");
      }
    }
  }
);
Custom registers override built-in widget types when the names match. Custom pages can be local Flutter pages and do not need to come from the RemUI server.

Variables

Set backend runtime variable

setVar('index', '0');

Set variable from callback response

const callback = ui.getCallback(req, res, '#cb1001');
callback
  .setVar('index', '1')
  .send();

Set a variable from a button tap

Use varSet() to produce an inline action node that updates a runtime variable on the client without a server round-trip. Bind the result to action, onPressed, or onTap.

ui('FilledButton', {
  action: varSet('index=1'),
  child: ui('Text', { text: 'Go to second tab' })
});

// Object form when the value isn't a string
ui('OutlinedButton', {
  action: varSet({ var: 'rememberMe', value: true }),
  child: ui('Text', { text: 'Remember me' })
});

The produced JSON shape is { "type": "setVar", "var": "index", "value": "1" }. The Flutter runtime updates the variable, increments the reactive tick, and any condition / templated Text bound to that variable rebuilds immediately.

Inline action object (no helper)

If you prefer an explicit JSON object, you can write the same payload by hand. This is what the sidebar and bottom-nav demos use:

ui('Sidebar', {
  variable: 'index',
  items: [
    {
      icon: ui('Icon', { icon: 'home' }),
      label: 'Form',
      action: { type: 'setVar', var: 'index', value: '0' }
    },
    {
      icon: ui('Icon', { icon: 'person' }),
      label: 'Profile',
      action: { type: 'setVar', var: 'index', value: '1' }
    }
  ]
});

Inline Actions

Inline actions are JSON nodes that the client executes immediately on tap, without contacting the server. They’re cheaper than a callback round-trip and are the right tool for local UI state changes (selected tab, active panel, theme toggle, locally-persisted preferences).

The available inline action types:

  • { type: 'setVar', var, value } — update a runtime variable
  • { type: 'setPrefs', entries: { ... } } — write entries to SharedPreferences and sync the matching prefs.* runtime variables
  • { type: 'nav', path, mode?, replace? } — navigate to a new page or open a dialog
  • 'nav:/some/path' / 'navReplace:/some/path' — shorthand string forms used by the nav() / navReplace() helpers

Persist preferences from a button tap

setPrefs() returns a setPrefs action node when used as an action prop. The client writes each entry to SharedPreferences and updates prefs.<key> / prefs.<key>.isPresent runtime vars so any bound UI reflects the change.

ui('FilledButton', {
  action: setPrefs({ theme: 'dark', lang: 'en' }),
  child: ui('Text', { text: 'Save preferences' })
});

// Querystring form
ui('OutlinedButton', {
  action: setPrefs('theme=dark&lang=en'),
  child: ui('Text', { text: 'Save preferences' })
});
An inline setPrefs action runs entirely on the client. It does not change the backend’s runtimeVars; the next page load will still hydrate from the request body. If you need both a client-side write and a server-side acknowledgement, use the callback form instead (see Callbacks).

Navigate from a button tap

// Push a new route on top of the current stack — back returns to the previous page.
ui('IconButton', {
  icon: ui('Icon', { icon: 'search' }),
  onPressed: nav('/ui/search')
});

// Replace the entire stack — the new page becomes the only route, back navigation is impossible.
ui('FilledButton', {
  action: navReplace('/ui/main'),
  child: ui('Text', { text: 'Sign in' })
});

// Open as a dialog overlay on top of the current page.
ui('IconButton', {
  icon: ui('Icon', { icon: 'open_in_new' }),
  onPressed: nav('/ui/dialogTest', 'dialog')
});

navReplace uses pushAndRemoveUntil internally so the entire route history is cleared. Use it for sign-in → home transitions or any flow where back navigation should not return to the prior screen.

Conditions

Generic condition

equate('index', '0', {
  child: ui('Text', { text: 'Visible when index == 0' })
});

Config value condition (eql)

ui('Text', {
  text: eql('showGreeting', 'true', 'Hello!')
    .equate('showGreeting', 'false', 'Goodbye')
    .else('…')
});

Produced JSON shape

{
  "type": "condition",
  "var": "index",
  "eq": "0",
  "child": { "type": "Text", "text": "..." }
}

Else If Chains

Use equate(...).equate(...) to build a chain that behaves like if / else if / else if. The first match is rendered and the rest are skipped.

equate('index', '0', {
  child: ui('Text', { text: 'Zero' })
}).equate('index', '1', {
  child: ui('Text', { text: 'One' })
}).equate('index', '2', {
  child: ui('Text', { text: 'Two' })
});

Callbacks

Callbacks are id-based and handled via ui.getCallback(req, res, id).

app.post('/ui/callbacks', (req, res) => {
  const callback = ui.getCallback(req, res, '#cb1001');
  const data = callback.data();

  callback
    .add(SnackBar('Submitted'))
    .setSharedPref('token=hello')
    .setPrefs({ theme: 'dark', lang: 'en' })
    .remSharedPref('token')
    .setVar('index', '1')
    .push('/ui/main')
    .pushReplace('/ui/home')
    .pushPage('/ui/main')
    .pushPageReplace('/ui/home')
    .send();
});

push / pushPage add a new route on top of the current stack. pushReplace / pushPageReplace clear the entire stack so the new route becomes the only one — back navigation will not return to any prior route.

sendBack timeout payload builder

const delayed = remui
  .sendBack()
  .timeout(1000)
  .setVar('searchDelayed', 'ready')
  .setSharedPref('searchDelayed=ready')
  .remSharedPref('dialogEmail');

Use this payload directly in the Timeout widget:

ui('Timeout', {
  data: delayed
});

Metadata and SSR

Every widget node exposes chainable metadata helpers. Use .meta() to define page metadata, and .tag() / .alt() to provide SSR-friendly HTML hints.

const screen = ui('Scaffold', {
  body: ui('Text', { text: 'Hello' })
})
  .meta('title', 'Hello Page')
  .meta('description', 'Rendered on the server too');

Server-side rendering is enabled with app.use(remui.ssr()). The middleware inspects the request and returns HTML for browser GET requests while Flutter keeps receiving JSON.

Flutter requests send a dedicated user-agent, RemUI-Flutter/1.0, so the middleware can distinguish app traffic from browser traffic.

Use $token$ inside Foreach templates and {token} for normal server/runtime interpolation. That keeps repeated row data separate from global variable substitution.

Submit Flow

Button action payload

ui('FilledButton', {
  action: {
    action: 'submit',
    id: '#cb1001',
    variables: ['name', 'password']
  },
  child: ui('Text', { text: 'Submit' })
});

Flutter sends id and selected variables to /ui/callbacks.

Controls

The Flutter widget registry is now split into smaller files under remx/lib/plugin/registers/. Widgets like Tabs, Radio, Checkbox, Slider, DatePicker, and Timeout live there so the registry can grow without staying monolithic.

See the widget reference for field-by-field schema details and examples.

Preference Conditions

Client sends preference presence in request context as:

{
  "prefs": {
    "token": { "isPresent": true, "value": "abc123" }
  }
}

Then backend can evaluate:

equate('prefs.token.isPresent', 'true', {
  child: ui('Text', { text: 'Token exists' })
});

And direct value reads work too:

ui('Text', { text: 'Token: {prefs.token}' })

Exceptional Widgets

These are widgets or behaviors that do not behave like ordinary plain JSON wrappers and need special handling in the runtime.

condition Renders conditionally and supports chained elseIf branches.
Text Supports template interpolation. Missing values keep their placeholder text so Foreach can decide whether to show or hide.
TextField Holds live state in runtime variables and can submit through the shared callback path.
Icon Backend resolves icon names to Material codepoints before the JSON is sent.
FilledButton / ElevatedButton / OutlinedButton These delegate click handling to the shared click dispatcher rather than custom per-widget logic.
Dialog Dialog pages are opened with nav('/ui/path', 'dialog') and stay layered over the current page.
Sidebar / NavigationRail Both are variable-reactive navigation widgets and can infer active state from setVar actions.
Foreach Supports remote requests, inline JSON variables, live updates on variable changes, and $token$ row interpolation.
callbacks Root-level callbacks are executed after the page loads, not during widget build.

File Layout

  • remui.ts: backend DSL, callback builder, icon resolver, variable runtime
  • index.ts: route definitions and callback handlers
  • remx/lib/plugin/remui.dart: Flutter transport and runtime state
  • remx/lib/plugin/clicks.dart: action execution and callback application
  • remx/lib/plugin/registers.dart: JSON widget registry and condition renderer
  • remx/lib/plugin/registers/: modular registry parts for feature-specific widgets
